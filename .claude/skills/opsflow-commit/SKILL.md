# opsflow-commit 技能

自动整理 git 工作区，清理乱码文件，生成贴合改动的提交注释，提交代码，并在项目根目录生成详细的 COMMIT_ANALYSIS.md 分析文档。

## 工作流程

### Step 1: 检查 git 状态

```bash
git status
git diff --stat
```

确认这是一个 git 仓库。如果不是，报错退出。

### Step 2: 清理乱码文件

乱码文件指的是**不应该是项目一部分**的垃圾文件，特征包括：

| 特征 | 示例 | 原因 |
|------|------|------|
| 空文件（0 字节）且位于项目根 | `32` | shell 输出重定向残留 |
| 文件名含 `(`, `)`, `{`, `}`, `%` | `({`, `0)`, `applyNodeColor(cell.id` | 命令/代码片段被错误写成文件名 |
| 文件名像 git diff 输出片段 | `400)`, `backend/%d` | git 命令输出残留 |
| 文件名是单行代码片段 | `apply`, `n.x` | 复制粘贴时意外创建 |

**清理方法：**

对每个乱码文件：
1. 用 `file <path>` 或 `ls -la <path>` 确认它确实是垃圾文件（不是项目中的真实文件）
2. 删除它们：
   - 未被 git 追踪：直接用 `rm -f <path>`
   - 已被 git 追踪：用 `git rm --cached <path>`（保留 git 历史记录）
3. **如果遇到安全权限确认（"Are you sure?" 或自动安全模式阻止删除）**：解释这个文件是空的/无用的，请求允许继续。必要时用 `rm -f` 直接删除而不是 `git rm`。

> **关于安全模式：** 删除文件可能会触发权限确认。如果被阻止，向用户说明"这是一个空的垃圾文件（0 字节），文件名是操作残留，不是项目代码"，请求通过。如果删除被拒绝，跳过该文件继续其他步骤。

### Step 3: 暂存改动

```bash
git add -A
```

然后检查是否有合理的变更要被提交：

```bash
git diff --cached --stat
```

如果没有任何变更，报错退出并提示"没有可提交的变更"。

### Step 4: 分析 diff 并生成提交注释

运行 `git diff --cached` 获取完整 diff 内容，分析所有改动的文件：

1. **分类总结**每个改动是什么（新功能/修复/重构/优化/文档/配置/依赖更新）
2. **为每个改动写一行概要**（中英文混合，用中文描述具体内容）
3. **组合成多行 commit message**

**Commit message 格式要求：**

```
<类型>: <英文概要> — <中文描述>

## Changes (中文)

- <文件名>: <具体改动说明>
- <文件名>: <具体改动说明>
```

类型前缀：`feat`(新功能), `fix`(修复), `refactor`(重构), `perf`(优化), `docs`(文档), `chore`(杂项/配置/依赖), `style`(样式), `revert`(回退)

Commit message 用英文写类型和概要，中文写详细描述。**这是写入 git log 的历史记录，要清晰且专业。**

### Step 5: 提交

```bash
git commit -m "<commit message>"
```

记录返回的 commit hash。

### Step 6: 追加 COMMIT_ANALYSIS.md（累积模式，不覆盖）

`COMMIT_ANALYSIS.md` 是**累积的提交记录日志**，每次提交在前面追加新的分析，旧的记录保留在后面。

**操作流程：**
1. 先生成本次提交的分析条目（每条的格式见下方）
2. 检查 `COMMIT_ANALYSIS.md` 是否已存在
3. **如果文件不存在**：直接用分析内容创建一个全新的文件（含标题 + 首条记录）
4. **如果文件已存在**：将本次的分析条目**插入到第一个 `---` 分隔线之后、第二条记录之前**（即追加到最前面，保持时间倒序）

**本次提交的分析条目格式：**

```markdown
## <Commit hash>

> 提交日期: <日期> | 提交信息: <commit message 第一行>

### 改动

| 文件 | 类型 | 说明 |
|------|------|------|
| `<file>` | 前端/后端/文档/配置 | <改动概述> |

### 解决

- **问题/背景：** <简要说明为什么有这个改动>
- **办法：** <如何解决>

### 验证

- 改动类型: feat/fix/refactor/...
- 清理乱码: 有/无
- 工作区状态: 干净 ✅
```

**文件结构（追加后效果）：**

```markdown
# Commit Analysis Log

<!-- 每次提交在最前面插入新条目，时间倒序排列 -->

## <最新 commit hash>

...最新提交的分析...

---

## <前一次 commit hash>

...前一次的分析...

---

## <第一次 commit hash>

...第一次的分析...
```

> **注意：** `COMMIT_ANALYSIS.md` 生成后不要 `git add` 它（它是独立于 commit 的分析记录，不属于任何单一提交）。

告诉用户：
- 提交的 commit hash
- 文件改动统计（几个文件，多少行变更）
- COMMIT_ANALYSIS.md 已生成
- 是否有任何乱码文件被清理

## 注意事项

1. **不要提交** `.env`, `*.pyc`, `node_modules/`, 密钥文件等
2. 如果 `.gitignore` 中已有忽略规则，`git add -A` 会自动跳过
3. 不要在 commit message 中添加 `Co-Authored-By` 等尾部标记，除非项目有明确要求
4. 如果 diff 很大（>200 行），不要在 commit message 中逐行描述，而是概括文件和改动类型
5. `COMMIT_ANALYSIS.md` 是累积日志，不要 `git add` 它（不属于任何单一提交，独立于版本控制之外）
