# Debug Doc Writer

将当前对话中的调试排障过程整理成结构化文档，保存到 `docs/<app>/debug/YYYY-MM-DD-<issue-name>.md`。

## 工作流程

### Step 1: 确定目标 App 和 issue 名称

从对话中最近修改的文件路径推断目标 App：

| 路径模式 | App 目录 |
|---------|---------|
| `backend/opsflow/` 或 `web/.../opsflow/` | `docs/opsflow/debug/` |
| `backend/cmdb/` 或 `web/.../cmdb/` | `docs/cmdb/debug/` |
| `backend/itsm/` 或 `web/.../itsm/` | `docs/itsm/debug/` |
| 其他 App 类似 | 对应 `docs/<app>/debug/` |

issue 名称从对话主题中提取，用 kebab-case 格式（如 `exclusive-gateway-condition-evaluation`）。

**输出路径：** `docs/<app>/debug/YYYY-MM-DD-<issue-name>.md`

如果 `docs/<app>/debug/` 目录不存在则自动创建。

### Step 2: 扫描对话提取内容

阅读当前对话历史（包括用户提问、错误日志、代码 diff、你的回答），提取以下信息：

1. **背景 & 现象** — 用户遇到了什么问题？有什么具体表现？（错误信息、异常日志、UI 截图等）
2. **排查思路** — 排查时看了哪些文件？做了哪些实验？排除了哪些可能性？
3. **根因分析** — 问题的根本原因是什么？（代码 bug、架构缺陷、配置错误、时序问题等）
4. **解决方案** — 具体改了哪些文件？改了什么？每一处修改的目的是什么？
5. **引入的方法 / 函数 / 设计模式** — 这次修复中新增了哪些函数、类、接口、设计模式？
6. **验证** — 如何确认修复有效？（测试、日志、UI 验证等）

### Step 3: 生成 MD 文档

参考 `docs/opsflow/debug/node-coloring-debug.md` 的格式风格：

```markdown
# <文档标题>

> 最后更新: YYYY-MM-DD
> 涉及版本: <如有>

---

## 1. 背景与现象

<描述用户遇到的原始问题和表现>

**错误日志 / 截图：**
```log
<关键错误信息>
```

## 2. 排查思路

<分步骤描述排查过程>

### 2.1 初步分析
### 2.2 <后续步骤>
### 2.n <更多步骤>

## 3. 根因分析

<详细解释问题的根本原因，可配合 ASCII 图示>

## 4. 解决方案

<按文件列出所有改动>

### 4.1 <文件名>

- **改动：** <改了什么>
- **目的：** <为什么这样改>

### 4.2 <文件名>
...

## 5. 引入的方法 / 函数 / 设计模式

| 引入内容 | 所在文件 | 说明 |
|---------|---------|------|
| `<函数/类名>` | `<文件路径>` | `<作用描述>` |
| `<设计模式>` | `<文件路径>` | `<应用场景>` |

## 6. 验证

<如何验证修复有效>

## 7. 涉及文件清单

- `<文件路径>` — `<改动概述>`
```

### Step 4: 写入文件

1. 确认输出目录存在（不存在则创建）
2. 写入 MD 文件
3. 确认文件写入成功
4. 告知用户文件路径

## 重要提示

- 日期格式统一为 `YYYY-MM-DD`
- 代码块标注语言（`python`、`typescript`、`log`、`bash` 等）
- ASCII 图示使用 `┌─┐` 等字符，保持对齐
- 涉及的文件路径使用相对路径（从项目根开始）
- 函数/类名使用代码引用格式 `` `func_name()` `` 或 `` `ClassName` ``
- commit hash 使用短格式（前 8 位）
- 文档保持在 500 行以内
