# 交易日志与错误监控功能说明

## 功能概述

交易日志和错误日志模块用于记录策略运行过程中的关键操作和异常信息，支持按时间、函数名、日志级别、合约代码等维度检索，是系统运维和策略排查的核心工具。

---

## 模块划分

| 模块 | 用途 | 数据来源 |
|------|------|---------|
| **交易日志 (TradeLog)** | 记录正常业务流程的关键节点 | 主动记录（决策过程、计算结果、API调用） |
| **错误日志 (ErrorLog)** | 记录异常和错误 | 被动捕获（try/except 异常堆栈） |

---

## 交易日志

### 字段说明

| 字段 | 说明 | 示例 |
|------|------|------|
| `timestamp` | 日志时间 | 2026-05-08 15:02:33 |
| `function_name` | 函数完整路径 | `stock.scheduler.tasks_daily_open.execute_entry_order` |
| `log_level` | 日志级别 | INFO / SUCCESS / WARNING |
| `symbol` | 关联合约代码 | rb2610 |
| `log_message` | 日志内容 | "rb2610 开仓成功: 1 Unit(4手) @ 3200.00" |

### 日志级别定义

| 级别 | 使用场景 |
|------|---------|
| `INFO` | 一般流程记录（如开始执行、参数值） |
| `SUCCESS` | 操作成功（如开仓成功、止损平仓成功） |
| `WARNING` | 需要关注但不致命（如跳空保护跳过、两步开仓） |
| `ERROR` | 错误（ TradeLog 中较少使用，通常写入 ErrorLog） |

### 典型日志内容

```
[INFO]  开始执行收盘计算任务
[INFO]  rb2610 趋势因子: 0.357, 标签: weak_bull
[SUCCESS] rb2610 开仓成功: 1 Unit(4手) @ 3200.00
[WARNING] m2609 跳空幅度过大，禁止开仓
[INFO]  今日共生成 3 个开仓信号, 2 个止损信号
```

## 错误日志

### 字段说明

| 字段 | 说明 | 示例 |
|------|------|------|
| `timestamp` | 错误发生时间 | 2026-05-08 15:02:45 |
| `function_name` | 出错函数完整路径 | `stock.scheduler.tasks_daily_close.update_all_positions_stop_loss_price` |
| `error_message` | 错误详情和堆栈 | Traceback + 上下文信息 |

### 典型错误场景

| 错误 | 常见原因 | 处理建议 |
|------|---------|---------|
| 指标计算失败 | TqApi 数据不足、网络异常 | 检查 K 线数据长度，重试或跳过 |
| 止损更新失败 | highest_close 为 None | 初始化持仓跟踪价格 |
| 开仓失败 | 不在交易时间、TargetPosTask 超时 | 检查交易状态，手动补单 |
| 平仓失败 | 持仓已为空、API 连接中断 | 检查实盘持仓，手动平仓 |

## 错误邮件通知

### 概述

关键错误（止损执行失败、API 反复断连、信号执行异常等）在写入 `ErrorLog` 后，通过 `send_email_task` 自动推送到用户邮箱，避免需要主动查询才发现问题。

### 触发机制

`log_error()` 增加 `notify=False` 参数，调用方传入 `notify=True` 时触发邮件发送：

```python
from stock.utils.log_util import log_error

# 仅写库，不通知
log_error('my_function', '普通错误')

# 写库 + 邮件通知
log_error('my_function', '关键错误', notify=True)
```

### 通知规则

| 场景 | 收件人 | 优先级 |
|------|--------|--------|
| 止损执行失败 | 账户所属用户的邮箱 | 高 |
| API 断连 | 同上 | 高 |
| 开仓/加仓/平仓/移仓异常 | 同上 | 中 |
| 持仓校验差异 | 下方默认邮箱 | 低 |

收件人逻辑：优先使用 `TradingAccount.user.email`，无配置时默认发送到 `312711936@qq.com`。

### 邮件格式

- 主题：`[交易系统] {function_name} 发生错误`
- 正文：函数名 + 错误详情（前 2000 字符）

### 已接入 notify=True 的模块

| 模块 | 触发点 |
|------|--------|
| `order_signals.py`  | 加仓/开仓/平仓/移仓各阶段失败 |
| `stop_loss_executor.py` | 止损平仓超时、持仓止损检查失败 |
| `tasks_daily_open.py` | 开盘处理账户级异常 |

## 持仓校验（DB vs TqSDK）

### 概述

每日 15:35 自动比对数据库持仓记录（`PositionState`）与交易所实际持仓（通过 TqSDK `get_position()` 获取），发现差异通过邮件通知用户手动处理。**仅检测，不自动修复**。

### 执行流程

```
15:32  盘后指标计算（结算价已确定）
15:35  持仓校验任务触发
  ├─ 获取第一个活跃账户的 TqApi 连接
  ├─ 跳过非交易日
  ├─ 遍历每个活跃账户
  │   ├─ 读取 DB 中 units>0 的持仓
  │   ├─ 调用 TqSDK get_position() 获取实盘持仓
  │   ├─ 对比逐一检测差异
  │   └─ 记录差异到 TradeLog
  └─ 有差异 → 发送邮件报告
```

### 差异类型

| 类型 | 含义 | 严重程度 |
|------|------|---------|
| `DB_HAS_TQ_HAS_NOT` | DB 有持仓，但交易所无此持仓 | 高 |
| `DIRECTION_MISMATCH` | DB 与交易所方向不一致 | 高 |
| `VOLUME_MISMATCH` | DB 与交易所手数不一致 | 中 |
| `TQ_HAS_DB_HAS_NOT` | 交易所有持仓，但 DB 无记录 | 中 |
| `CHECK_ERROR` | 校验过程异常 | 低 |

### 邮件报告格式

主题：`[持仓校验] 发现 N 项差异`

HTML 表格报告包含：合约代码、差异类型、DB 手数、交易所手数。按账户分组展示。

### 依赖配置

- 定时任务：APScheduler `job_daily_reconciliation`，15:35 周一至周五
- Redis 分布式锁：`lock:daily_reconciliation` 防止并发
- 邮件发送：复用 `stock.tasks.send_mail.send_email_task`
