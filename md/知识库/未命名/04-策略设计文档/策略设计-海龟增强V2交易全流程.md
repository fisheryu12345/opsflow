## 一、开仓逻辑（ENTRY）

### 1.1 触发条件

```python
def check_breakout_signal(klines, entry_period=20):
    # 取前 entry_period 根 K 线的高低点（不含最新一根）
    high_prior = klines["high"].iloc[-entry_period - 1:-1]
    low_prior = klines["low"].iloc[-entry_period - 1:-1]
    
    entry_high = float(high_prior.max())
    entry_low = float(low_prior.min())
    latest_close = float(klines.iloc[-1]['close'])
    
    if latest_close > entry_high:
        → 向上突破，signal_direction = 1（做多）
    elif latest_close < entry_low:
        → 向下突破，signal_direction = -1（做空）
```

**关键设计原则**：
- 入场**只看 20HL 通道突破**，不参考 MA 排列方向
- MA 10/20/40 排列**仅用于趋势因子（trend_factor）计算**，进而动态调整止损距离：强趋势→止损放宽（3.0 ATR），震荡→止损收紧（2.0 ATR）
- 这种设计的目的是：不错过任何趋势启动，用止损宽度来控制风险而非用过滤机制

**关键规则**：
- 突破判断使用**前 20 日的高低点，不包含当日 K 线**
- 收盘后计算，次日开盘执行
- 信号生成时检查：
  1. 该品种当前 `units == 0`（无持仓）
  2. 不存在未执行的 `ENTRY` 信号（`executed_status='PENDING'`）
  3. 该品种在当前账户的 `AccountContractConfig` 中已激活（`is_active=True`）

### 1.2 仓位计算

```python
def calculate_unit_lots(api, symbol):
    contract = api.get_quote(symbol)
    volume_multiple = contract.volume_multiple  # 合约乘数
    
    # 计算 20 日 ATR
    klines = api.get_kline_serial(symbol, duration_seconds=86400, data_length=25)
    # ... 计算 TR 列表，取最近 20 个 TR 的平均值 → atr_20
    
    unit_lots = POSITION_RISK_BASE_AMOUNT / (atr_20 * POSITION_RISK_MULTIPLIER * volume_multiple)
    unit_lots = round(unit_lots)
    unit_lots = max(1, unit_lots)
    return unit_lots
```

**公式**：`unit_lots = 4000 / (ATR × 2 × 合约乘数)`

**示例**：螺纹钢 ATR=50，乘数=10 → `unit_lots = 4000 / (50×2×10) = 4 手/Unit`

### 1.3 跳空保护（方向区分）

```python
def price_gap_protection(api, symbol, direction, gap_threshold_atr_multiplier=1.5):
    atr = calculate_atr(api, symbol)
    quote = api.get_quote(symbol)
    
    if direction == 1:  # 多头
        gap_up = (latest_price - pre_close) / atr
        if gap_up > gap_threshold_atr_multiplier: → 禁止开仓
    elif direction == -1:  # 空头
        gap_down = (pre_close - latest_price) / atr
        if gap_down > gap_threshold_atr_multiplier: → 禁止开仓
```

> **方向区分逻辑**：多头只拦截跳空高开（买入价过高），空头只拦截跳空低开（卖出价过低）。反向跳空（多头遇低开、空头遇高开）视为有利，不拦截。
>
> 跳空阈值 = 1.5 倍 ATR，防止隔夜重大跳空导致的追高/追低。

### 1.4 最小开仓手数限制（两步开仓）

当 `planned_volume < min_position`（交易所最小开仓手数）时：

```
第1步：开立 min_position 手（满足交易所要求）
第2步：立即平仓 (min_position - planned_volume) 手
结果：最终持仓 = planned_volume 手，合规且达到目标
```

> 例如：计划开 2 手，但交易所最小开仓 5 手 → 先开 5 手，再平 3 手，最终持仓 2 手。

### 1.5 开仓执行流程

```python
def execute_entry_order(api, account, signal):
    1. 检查交易时间 is_trading()
    2. 跳过震荡行情检查（skip_choppy_entry=True 且趋势为 choppy/neutral 时取消）
    3. 计算 unit_lots → order_volume = 1 * unit_lots
    4. 跳空保护检查
    5. 最小开仓手数检查
    6. TargetPosTask 设置目标持仓
       - 多头：target_lots = +order_volume
       - 空头：target_lots = -order_volume
    7. wait_for_target_position(timeout=60s)
    8. 成交后更新 PositionState：
       - units = 1
       - direction = signal_direction
       - contract_total_position = actual_filled
       - first_open_price = entry_avg_price
       - highest_close = entry_avg_price
       - lowest_close = entry_avg_price
       - open_date = today
       - entry_trend_factor = signal.trend_factor   ← 新：入场趋势因子
       - entry_trend_label = signal.trend_label      ← 新：入场趋势标签
       - entry_atr = atr_20                          ← 新：入场 ATR
    9. 信号状态更新为 SUCCESS
```

### 1.6 震荡行情跳过开仓（A/B 对比验证）

当 `execute_entry_order` 检测到账户策略配置 `skip_choppy_entry = True` 时，对趋势标签为 `choppy` 或 `neutral` 的信号直接取消，不开仓。

```python
# execute_entry_order 中，is_trading 检查之后、ATR 计算之前
try:
    config = account.strategyconfig
    if config.skip_choppy_entry and signal.trend_label in ('choppy', 'neutral'):
        signal.executed_status = 'CANCELLED'
        signal.remark = '震荡行情不入场'
        signal.save(update_fields=['executed_status', 'updated_at', 'remark'])
        return False
except StrategyConfig.DoesNotExist:
    pass
```

**设计意图**：
- 用于 **A/B 对比验证**：A 账户 `skip_choppy_entry = False`（正常开仓），B 账户 `skip_choppy_entry = True`（跳过震荡）
- 目的是验证"震荡行情开仓是否值得"——若 B 的绩效长期优于 A，说明在震荡行情中强开不如不开
- 此开关不影响止损行为，只控制开仓入口

**备注对照**：

| 趋势标签 | skip_choppy_entry=False | skip_choppy_entry=True |
|---------|------------------------|------------------------|
| strong_bull / strong_bear | 正常开仓 | 正常开仓 |
| weak_bull / weak_bear | 正常开仓 | 正常开仓 |
| choppy / neutral | 正常开仓 | **取消（备注：震荡行情不入场）** |

### 2.2 临收盘紧急止损（14:55）

```python
def check_and_execute_stop_loss(api):
    # 用 TqSDK 实时行情判断触发止损
    for position in 所有有持仓:
        quote = api.get_quote(position.symbol)
    api.wait_update(10s)
    
    for position in positions:
        # ★ 实时止损重算逻辑（2026-05-14 新增）
        # DB中 stop_loss_price 是昨日15:32计算的，经过一天行情可能已过时
        # 改用实时 ATR + trend_factor 重新计算止损价
        indicators = calculate_indicators(api, position.symbol)
        if indicators and indicators['atr_20']:
            fresh_atr = indicators['atr_20']
            fresh_factor = indicators['trend_factor']
            if 多头:
                recalc_stop = highest_close - 2*(1+fresh_factor)*fresh_atr
            if 空头:
                recalc_stop = lowest_close + 2*(1+fresh_factor)*fresh_atr
        # 重算失败则回退到DB原止损价
        final_stop = recalc_stop if recalc_stop else stop_loss_price
        
        if 多头 and quote.last_price < final_stop:
            → 触发
        if 空头 and quote.last_price > final_stop:
            → 触发

    for position in positions:
        1. 检查是否存在未执行的 STOP_LOSS 信号（防重复）
        2. 创建 STOP_LOSS 信号（executed_status='EXECUTING'）
        3. TargetPosTask 设置目标持仓 = 0（市价平仓）
        4. 等待成交（timeout=60s）
        5. 创建 ClosedPositionRecord
        6. 清空 PositionState（units=0, direction=0, stop_loss_price=None...）
        7. 更新信号状态为 SUCCESS
```

> **14:55 与开盘止损的区别**：14:55 是紧急风控，实时重算止损后触发即平；开盘止损是按信号队列顺序执行。

### 2.4 平仓趋势记录

无论哪种平仓路径，最终的 `record_and_reset_position()` 或 `execute_rollover_order()` 都会在 `ClosedPositionRecord` 中写入以下趋势分析字段：

| 数据 | 来源 | 说明 |
|------|------|------|
| `entry_trend_factor/label/atr` | 从 `PositionState.entry_*` 直接复制 | 入场时的市场状态快照，开仓时冻结 |
| `exit_trend_factor/label` | 从平仓信号 `signal.trend_*` 读取 | 出场时的趋势状态 |
| `exit_atr` | 实时调用 `calculate_atr()` 计算 | 出场时的波动率 |
| `max_favorable_excursion` | `position.highest_close - cost_price`（多头） | 持仓期间最大有利价格偏移（MFE） |
| `max_adverse_excursion` | `cost_price - position.lowest_close`（多头） | 持仓期间最大不利价格偏移（MAE） |

```python
def record_and_reset_position(api, position, signal, filled_volume, avg_price):
    # ... (原有 PnL 计算) ...

    # 出场趋势：信号优先，降级到 position.indicators
    if signal is not None:
        exit_trend_factor = signal.trend_factor
        exit_trend_label = signal.trend_label
    elif position.indicators:
        exit_trend_factor = position.indicators.get('trend_factor', 0)
        exit_trend_label = position.indicators.get('trend_label', '')

    # 出场 ATR
    exit_atr = calculate_atr(api, position.symbol)

    # MFE/MAE
    if direction == 1:  # 多头
        mfe = position.highest_close - cost_price
        mae = cost_price - position.lowest_close
    else:               # 空头
        mfe = cost_price - position.lowest_close
        mae = position.highest_close - cost_price

    ClosedPositionRecord.objects.create(
        ...原有字段...,
        entry_trend_factor=position.entry_trend_factor,
        entry_trend_label=position.entry_trend_label,
        entry_atr=position.entry_atr,
        exit_trend_factor=exit_trend_factor,
        exit_trend_label=exit_trend_label,
        exit_atr=exit_atr,
        max_favorable_excursion=mfe,
        max_adverse_excursion=mae,
    )
```

### 2.5 数据关联示例如下

```
开仓 (execute_entry_order)
  └─→ signal.trend_factor/label + atr_20 → PositionState.entry_*
                                   (写入一次，永不修改)

每日收盘 (update_all_positions_high_low_price)
  └─→ 现在同时跟踪 highest_close 和 lowest_close（双向）
      使 MFE/MAE 对所有持仓方向都有意义

平仓 (record_and_reset_position / execute_rollover_order)
  ├─ PositionState.entry_*        → ClosedPositionRecord.entry_*
  ├─ signal.trend_factor/label    → ClosedPositionRecord.exit_*
  ├─ calculate_atr()              → ClosedPositionRecord.exit_atr
  └─ highest_close/lowest_close   → ClosedPositionRecord.mfe/mae
```

### 3.2 移仓执行流程

```python
def execute_rollover_order(api, position, signal):
    # ========== 第1阶段：平仓旧合约 ==========
    target_pos_old = TargetPosTask(api, position.symbol)
    target_pos_old.set_target_volume(0)
    wait_for_target_position(...)
    
    # ========== 第2阶段：开仓新合约 ==========
    target_volume = position.contract_total_position  # 同方向同手数
    
    # 从 FullContractList 获取新主力合约代码
    main_contract = FullContractList.objects.filter(
        product_code=position.product_code
    ).first()
    new_symbol = main_contract.symbol
    
    # 检查最小开仓限制
    min_position_check = check_min_position_requirement(new_symbol, target_volume)
    if min_position_check['need_adjustment']:
        → 两步开仓 (使用 new_symbol)
    else:
        target_pos_new = TargetPosTask(api, new_symbol)
        target_pos_new.set_target_volume(target_lots)
        wait_for_target_position(...)
    
    # ========== 第3阶段：更新数据库 ==========
    # 新合约初始数据（一次 get_kline_serial 覆盖旧开仓至今的完整K线）
    holding_days = (today - position.open_date).days
    data_length = max(25, holding_days + 10)
    klines = api.get_kline_serial(new_symbol, duration_seconds=86400, data_length=data_length)

    # 完整历史最高/最低收盘价
    H_full = klines['close'].max()
    L_full = klines['close'].min()
    init_highest_close = H_full (多头) / None (空头)   # 跟踪止损参考值
    init_lowest_close = L_full (空头) / None (多头)

    # 新合约 20 日 ATR + 止损距离
    atr_val = calculate_atr(api, new_symbol, period=20)
    stop_dist = 2 × (1 + trend_factor) × atr_val     # trend_factor 复用旧合约信号

    # 旧合约单位盈利保本线兜底
    old_profit = abs(exit_price - cost_price)
    entry_dec = Decimal(entry_avg_price)
    多头: init_stop_loss = max(H_full - stop_dist, entry_dec - old_profit)
    空头: init_stop_loss = min(L_full + stop_dist, entry_dec + old_profit)

    # 创建新 PositionState
    PositionState.objects.create(
        account=position.account, symbol=new_symbol,
        direction=position.direction,
        units=position.units,
        highest_close=init_highest_close,
        lowest_close=init_lowest_close,
        stop_loss_price=init_stop_loss,
        protect_cost_enabled=旧持仓是否已激活保本,
        is_rollover_needed=False
    )
    # 删除旧合约 PositionState
    PositionState.objects.filter(id=position.id).delete()
```


> **优先级**：开盘任务中，ROLLOVER 在 ENTRY 之后、ADD_ON 之前执行。

### 3.3 初始止损计算公式分析

移仓后新合约的 `init_stop_loss` 公式（取代原有的 20 日 ATR 简单计算）：

**多头：** `init_stop_loss = max(H_full - stop_dist, entry - old_profit)`

**空头：** `init_stop_loss = min(L_full + stop_dist, entry + old_profit)`

| 符号 | 含义 | 来源 |
|------|------|------|
| H_full / L_full | 新合约自旧开仓日至今的最高/最低收盘价 | `get_kline_serial` 全量数据 |
| stop_dist | 标准止损距离 `2 × (1+tf) × ATR` | `calculate_atr` + 旧合约 trend_factor |
| entry | 新合约开仓均价 | Phase 2 执行结果 |
| old_profit | 旧合约单位盈利 `abs(exit_price - cost_price)` | Phase 1 平仓结果 |

**两重保护的数理逻辑：**

1. **第一项（H_full - stop_dist）**：相当于"如果新合约在历史最高收盘那天开仓，跟踪止损现在会在哪"。把新合约已走过的行情动量纳入止损位置。当新合约在此期间已经上涨，止损紧贴当前价；反之止损较宽。

2. **第二项（entry - old_profit）**：旧合约已实现盈利为 `old_profit × 手数`，新仓从 entry 开始每跌 1 点就亏 1 点，当新仓累计亏损 = 旧仓已实现盈利时，整笔操作归零。此价格即是底线保护。

3. **max/min 组合**：取更紧的保护。

| 新合约表现 | 第一项 | 第二项 | 生效项 | 效果 |
|-----------|--------|--------|--------|------|
| 已大涨（H_full >> entry） | 紧贴开仓价 | 远低于开仓价 | 第一项 | 利润已保护 |
| 未涨/微涨（H_full ≤ entry） | 较宽 | 兜底价格较高 | 第二项 | 整体保本 |

**与收盘任务的无缝衔接：**

`init_highest_close` 直接设为 `H_full`，移仓后第一次收盘任务执行时：
```
最高收盘 = max(H_full, 当日收盘) = H_full（除非创新高）
动态止损 = 最高收盘 - stop_distance = H_full - stop_distance ≈ init_stop_loss
```
移仓瞬间的初始止损和收盘任务的跟踪止损在公式上一致，没有跳变。

**trend_factor 说明：** 移仓时复用旧合约信号的 trend_factor，同品种均线排列类同偏差可忽略。收盘后首个收盘任务即使用新合约自身指标重新计算 trend_factor，此处只需一个合理初始值。

## 四、加仓逻辑（ADD_ON）

### 4.1 加仓信号生成（收盘后）

**今日优化（2026-05-15）：从方向分支到 `favorable_diff` 统一**

将多头/空头的重复分支合并为 `favorable_diff`（正数 = 价格朝有利方向移动），约 40 行精简为约 15 行。同时增加移仓换月重定向：老合约标记 `is_rollover_needed=True` 时，将 ADD_ON 信号切换到新主力合约。

```python
def check_add_position_signals():
    positions = PositionState.objects.filter(units__gt=0, units__lt=3)

    for position in positions:
        # 移仓换月检测：老合约禁止开仓，切换到新合约
        target_symbol = position.symbol
        target_product_code = position.product_code

        if position.is_rollover_needed:
            main_contract = FullContractList.objects.filter(
                product_code=position.product_code
            ).first()
            if main_contract and main_contract.symbol != position.symbol:
                target_symbol = main_contract.symbol
                target_product_code = main_contract.product_code
                # 信号 symbol 已切换为新合约
            else:
                continue  # 无法确定新合约，跳过

        atr_value = position.indicators['atr_20']
        latest_price = position.latest_close_price
        first_open_price = position.first_open_price  # 统一基准

        price_diff = latest_price - first_open_price  # 有符号，用于日志
        if position.direction == 1:
            favorable_diff = price_diff        # 多头：涨为正方向
        else:
            favorable_diff = first_open_price - latest_price  # 空头：跌为正方向

        if favorable_diff > 0.5 * atr_value:
            if current_units == 1:
                # 超 0.5×ATR → +1，超 1.0×ATR → +2（满仓）
                add_units = 2 if favorable_diff > 1.0 * atr_value else 1
            elif current_units == 2 and favorable_diff > 1.0 * atr_value:
                # (累计) 超 1.0×ATR → +1
                add_units = 1

        # 安全截断
        if current_units + add_units > 3:
            add_units = 3 - current_units

        # 防重复（用 target_symbol 检查）
        if 存在 PENDING 的 ADD_ON 信号(按 target_symbol): → 跳过

        DailyStrategySignal.objects.create(
            trade_type='ADD_ON',
            symbol=target_symbol,            # ← 移仓重定向时为新合约
            product_code=target_product_code, # ← 移仓重定向时为新合约
            contract_target_number=add_units,
            remark=基础备注 + (移仓重定向备注),
        )
```

> **前后变化：**
>
> | 维度 | 改前 | 改后 |
> |------|------|------|
> | 代码量 | ~40 行（units×direction 4 分支） | ~15 行（favorable_diff 统一） |
> | 计算 | 多头/空头各写一套 price_diff/条件 | 一次计算 favorable_diff = 正数 |
> | units=2 空头 price_diff 符号 | 正（`first_open - latest`） | 负（`latest - first_open`），仅影响日志 |
> | 移仓重定向 | 无 | `is_rollover_needed=True` → 信号发到新合约 |
> | 逻辑结果 | 完全相同 | 完全相同 |

> **为什么 units=2 空头的 price_diff 符号变化不影响策略？**
> 因为加仓判断使用 `favorable_diff`（正数），而 `price_diff` 仅用于 remark 日志。日志中的方向描述由文本 `"空头"` 保证，diff 数值符号不影响可读性。

### 4.2 加仓执行流程

```python
def execute_add_on_order(api, account, signal):
    1. 检查交易时间 is_trading()
    2. 从 signal.contract_target_number 获取加仓单位数
    3. 查找持仓：
       - 优先通过 symbol 查找 PositionState
       - 若旧合约已移仓（symbol 不存在），按 product_code 找新合约持仓
    4. 检查 projected_units = position.units + add_units > 3 ?
       → 是：取消信号（CANCELLED）
    5. 计算 order_volume = add_units * unit_lots
    6. 检查最小开仓限制
    7. TargetPosTask 设置目标持仓
       - 多头：target_lots = current_position + order_volume
       - 空头：target_lots = -(current_position + order_volume)
    8. wait_for_target_position(timeout=60s)
    9. 更新 PositionState：
       - units += add_units
       - contract_total_position += order_volume
    10. 信号状态更新为 SUCCESS
```

> **移仓回退**：如果加仓信号对应的旧合约持仓已被移仓删除，按 `product_code` 查找新合约的持仓继续执行加仓。防止移仓后加仓信号因找不到原持仓而失败。
>
> **移仓重定向（2026-05-15 新增）**：在收盘信号生成阶段，若持仓标记 `is_rollover_needed=True`，ADD_ON 信号的 `symbol` 直接使用新主力合约。开盘执行时 ROLLOVER（Step 3）先于 ADD_ON（Step 4），因此 ADD_ON 执行时新合约的 PositionState 已存在，无需回退查找。

### 4.3 加仓价格坐标推导（多头示例）

假设开仓价 P₀=5000，ATR=40：

```
价格轴（向上为正）:
                         价格
                          │
         第3加仓点 ──────┼────── 5040  ← 1Unit时涨超1.0A，+2Unit（直接满仓）
                          │              2Unit时累计涨超1.0A，+1Unit（满仓）
         第2加仓点 ──────┼────── 5020  ← 1Unit时涨超0.5A，+1Unit
                          │
         开仓点   ──────┼────── 5000  ← ENTRY（唐奇安突破）
                          │
                          ▼
```

**关键观察**：
- units=1 和 units=2 统一以 `first_open_price` 为基准，无基准切换
- 价格从 5000 → 5020：+1Unit（总 Unit=2）
- 价格从 5000 → 5040：+2Unit（总 Unit=3，直接满仓）
- 若先涨到 5020（2Unit），再涨到 5040：累计涨幅 = 40 = 1.0×ATR → 再触发 +1Unit 到满仓

### 5.1 动态跟踪止损

```python
def update_all_positions_stop_loss_price(api):
```

| factor | 止损倍数 | 特性 |
|--------|---------|------|
| 0.0 | 2.0 ATR | 震荡市，止损收紧 |
| 0.25 | 2.5 ATR | 中等趋势 |
| 0.5 | 3.0 ATR | 强趋势，给足回调空间 |

### 5.2 保本兜底机制

**启用条件**（两步限制）：
1. 持仓必须达到 **3 个 Unit（满仓）** 才允许激活
2. 浮动盈利 > `PROTECT_COST_ENABLED_RATIO × ATR`（默认 2.5 × ATR）

```python
# 保本启用条件（单向开关，只开不关）
if not protect_cost_enabled and cost_price and position.latest_close_price and position.units >= 3:
    if 浮动盈利 > 2.5 × ATR:
        protect_cost_enabled = True

# 保本价（按品种 tick size 动态计算）
tick = FullContractList.objects.get(symbol=position.symbol).price_tick
多头保本价 = 成本价 + tick × 2
空头保本价 = 成本价 - tick × 2

# 最终止损（保本只是底线）
多头：max(动态止损, 保本价)  # 止损不低于保本价
空头：min(动态止损, 保本价)  # 止损不高于保本价
```

> **重要设计**：`protect_cost_enabled` 是单向开关，一旦开启永不关闭。仅在 `units >= 3`（满仓）时才允许激活，避免加仓中途过早锁定利润导致趋势中过早离场。

### 5.3 止损触发检查（双保险）

| 检查时机 | 触发条件 | 执行方式 |
|---------|---------|---------|
| **15:02 收盘后** | 收盘价击穿止损价 | 生成 STOP_LOSS 信号，次日开盘执行 |
| **14:55 临收盘** | 最新价击穿止损价 | 立即市价平仓（紧急风控） |
| **09:02/21:02 开盘** | 执行 PENDING 的 STOP_LOSS 信号 | TargetPosTask 平仓 |

### 5.4 双向极值跟踪（新增）

为了保证平仓时 MFE/MAE 计算的完整性，`update_all_positions_high_low_price()` 已改为**双向跟踪**（原有逻辑只跟踪有利方向）：

```python
# 旧逻辑：多头只更新最高价，空头只更新最低价
if direction == 1 and 最新价 > highest_close:  # 只更新最高价
if direction == -1 and 最新价 < lowest_close:   # 只更新最低价

# 新逻辑：无论多空，同时跟踪两个极值
if 最新价 > highest_close:  # 更新最高价
    PositionState.objects.update(highest_close=最新价)
if 最新价 < lowest_close:   # 更新最低价
    PositionState.objects.update(lowest_close=最新价)
```

**为什么需要双向跟踪？**

以多头为例：开仓价 3500，价格先跌到 3450 再涨到 3600 平仓。
- 旧逻辑：`highest_close=3600` ✓，但 `lowest_close` 始终=3500（因为不更新），MAE=0
- 新逻辑：`highest_close=3600` ✓，`lowest_close=3450` ✓，MAE=-50

**对止损逻辑无影响**，因为：
- 多头止损只用 `highest_close`（新逻辑不会降低它）
- 空头止损只用 `lowest_close`（新逻辑不会抬高它）

## 六、信号防重复机制

```python
def check_duplicate_pending_signal(account, symbol, trade_type):
```

```
PENDING（待执行）

    ├── 开盘任务 pickup ──► EXECUTING（执行中）
```

## 十二、Alpha 因子分析

> 本文不分析实盘绩效，仅从策略设计角度拆解每个因子的数学逻辑、收益来源和适用环境。

### 12.1 因子全景

```
核心 Alpha
  ├── Donchian 20HL 突破 ───────── 趋势启动检测（~60-70% Alpha贡献）
  ├── ATR 头寸规模 ────────────── 波动率自适应风险分配（~15-20%）
  ├── ATR 动态止损 ───────────── 趋势跟踪止损优化（~10-15%）
  └── 金字塔加仓 ─────────────── 趋势确认放大收益（~10-15%）

信号质量增强
  ├── MA 趋势因子 ────────────── 趋势强度自适应止损宽度（边际改善 5-10%）
  └── Gap 保护 ──────────────── 避免高位接盘/低位砸盘（边际改善 3-5%）
```

### 12.2 因子详细拆解

#### 因子一：Donchian 20HL 突破（核心 Alpha 来源）

**定义**：`close > max(high[-21:-1])` → 做多；`close < min(low[-21:-1])` → 做空

**数学表达**：
```
signal = 
  +1  if close_t > max(high_{t-20}, ..., high_{t-1})
  -1  if close_t < min(low_{t-20}, ..., low_{t-1})
   0  otherwise
```

**收益来源**：趋势跟踪的基本逻辑——当价格突破过去 20 天的价格区间，意味着市场情绪发生质变（新买方/卖方进入市场）。趋势一旦形成，在短期内具有正向自相关性（momentum effect）。

| 属性 | 值 |
|------|-----|
| 因子类型 | 时间序列动量（Time-series Momentum） |
| 信号频率 | 低频（每品种每月约 1-3 次） |
| 预期胜率 | ~35-40% |
| 预期盈亏比 | ~2.5-3:1 |
| 最佳环境 | 趋势市场（单边上涨或下跌） |
| 最差环境 | 震荡市场（反复触发假突破） |

#### 因子二：ATR 头寸规模（风险平价因子）

**公式**：
```
unit_lots = max(1, round(4000 / (ATR × 2 × volume_multiple)))
```

**数理本质**：让 1 Unit 的名义风险在所有品种和所有时间点上保持一致。

| 参数 | 含义 | 效果 |
|------|------|------|
| `POSITION_RISK_BASE_AMOUNT = 4000` | 风险基数 | 每 Unit 承担约 2000 元风险（除以 multiplier 2） |
| 高 ATR → 手数少 | 波动率高时自动减仓 | 趋势末期波动率高峰→自动降低暴露 |
| 低 ATR → 手数多 | 波动率低时自动加码 | 趋势初期波动率低谷→自动增加暴露 |

**隐形的择时效果**：波动率在趋势末期膨胀、在趋势初期收缩是一个被广泛验证的市场现象（波动率聚集 + 均值回复）。ATR 头寸规模在这个意义上嵌入了一个**反脆弱的风险管理结构**：在风险最高时下注最小，在风险最低时下注最大。

**因子属性**：

| 属性 | 值 |
|------|-----|
| 因子类型 | 波动率管理（Volatility Targeting） |
| alpha 贡献方式 | 提升风险调整后收益（Sharpe），非直接预测价格方向 |
| 跨品种可比性 | ✅ 是（ATR 归一化） |
| 参数敏感性 | 低（round 取整后对参数小变化不敏感） |

#### 因子三：ATR 动态止损（趋势衰减因子）

**公式**（多头为例）：
```
stop_price = max(highest_close - 2 × (1 + trend_factor) × ATR, protect_price)
```

**核心机制**：不是在入场价设固定止损，而是**跟踪持仓期间的最高收盘价**，价格涨→止损上移，价格跌→止损不变。只有"从高点回落"才触发。

```
价格轴:
                    ▲ 最高收盘价
                   / \
                  /   \  止损距离 = 2×(1+factor)×ATR
                 /     \────────── ← 止损随最高价上移
                /                  ← 上涨全程不干预
               /                   ← 只有回落才触发
```

**这种行为模式的数学特征**：
- **路径依赖**：最终收益取决于价格路径中的极值，不是终值
- **正凸性**：在趋势行情中，止损不断上移锁定利润；在震荡中，止损距离收紧快速退出
- **非线性**：止损距离随趋势强度变化（2.0 ~ 3.0 ATR），实现状态依赖的风险管理

**与固定止损的根本区别**：

| 维度 | 固定百分比止损 | ATR 动态止损 |
|------|--------------|-------------|
| 止损位置 | 开仓价 ± 固定比例 | 最高价 - 动态距离 |
| 波动率适应性 | ❌ 不调整 | ✅ 自动适应 |
| 趋势适应性 | ❌ 不调整 | ✅ 趋势强→距离宽 |
| 盈利锁定 | ❌ 不锁定 | ✅ 最高价上移锁定 |
| 数学形态 | 线性 | 非线性 |

**因子属性**：

| 属性 | 值 |
|------|-----|
| 因子类型 | 路径依赖风险管理（Path-dependent Risk Management） |
| alpha 贡献方式 | 提升盈亏比（让盈利奔跑 + 截断亏损） |
| 对胜率的影响 | 降低（交易持有期更长，减少小额盈利交易） |
| 对盈亏比的影响 | 显著提升（放大盈利单，收紧亏损单） |

#### 因子四：金字塔加仓（动量确认因子）

**规则**：
```
current_units = 1:
  favorable_diff > 0.5×ATR → +1 Unit
  favorable_diff > 1.0×ATR → +2 Unit（满仓）
current_units = 2:
  cumulative_diff > 1.0×ATR → +1 Unit（满仓）
```

**数理逻辑**：这是"小仓位测试，确认后放大"的贝叶斯思维——先用 1/3 目标仓位入场，价格走势确认了你的判断后再追加。

**对比一次性满仓**：

| 场景 | 一次性满仓（1×3U） | 分批加仓（1U → 2U → 3U） |
|------|------------------|----------------------|
| 入场后立即反转 | 亏损 ×3 | 亏损 ×1（未加仓即止损） |
| 入场后小幅上涨后反转 | 盈亏平衡 | 小幅盈利后止损 |
| 入场后持续上涨 | 盈利 ×3 | 盈利 ×3（加仓到满） |
| 入场后大涨 | 盈利 ×3 | 盈利 ×3（加仓到满） |

**关键特征**：在不利场景中，分批加仓大比例降低了亏损——入场即反转时只亏 1/3。而在有利场景中，最终仓位也能达到满仓 3 单位，不牺牲上行收益。这是典型的不对称收益结构（positive convexity）。

**加仓基准选择**：V2 统一使用 `first_open_price` 为基准（非逐级递推），意味着加仓触发价完全由首次开仓价决定。

| 基准方式 | 公式 | 效果 |
|---------|------|------|
| 首次开仓价（V2） | `first_open + 0.5×ATR` | 整体平均成本上移较慢，加仓更激进 |
| 逐级递推 | `last_add + 0.5×ATR` | 整体平均成本上移较快，加仓更保守 |
| 首次+上次混合 | 依赖于具体计算 | 介于两者之间 |

**因子属性**：

| 属性 | 值 |
|------|-----|
| 因子类型 | 自适应执行（Adaptive Execution） |
| alpha 贡献方式 | 不对称收益结构——降低亏损场景暴露，保持盈利场景暴露 |
| 最佳环境 | 强趋势市场（加仓全部触发，满仓捕捉主趋势） |
| 最差环境 | 假突破 + 小幅反弹（加仓后立即止损，亏损放大） |

#### 因子五：MA 趋势因子（趋势强度自适应因子）

**公式**：
```
trend_strength = min(max(|MA10-MA20|, |MA20-MA40|) / ATR / 2.0, 1.0)
trend_factor = round(trend_strength × 0.5, 3)
```

> 趋势因子不直接产生交易信号，它是止损距离的调节器。详见 [指标-趋势因子.md](指标-趋势因子.md)。

**因子属性**：

| 属性 | 值 |
|------|-----|
| 因子类型 | 状态依赖调节（Regime-dependent Modulation） |
| 直接作用 | 调节止损宽度 [2.0, 3.0] ATR |
| alpha 贡献方式 | 间接——强趋势时给更多空间减少被洗出，弱趋势时收紧快速止损 |

#### 因子六：Gap 保护（反脆弱因子）

**定义**：当 `|open - pre_close| / ATR > 1.5` 时，顺向跳空方向禁止开仓。

**本质**：顺向跳空（多头遇到高开、空头遇到低开）意味着入场成本显著劣化。此因子不是预测价格方向，而是**避免以不利价格入场的执行优化**。

**因子属性**：

| 属性 | 值 |
|------|-----|
| 因子类型 | 执行优化（Execution Optimization） |
| alpha 贡献方式 | 避免极端入场价，提升整体盈亏比 |
| 触发频率 | 低（每品种每月约 0-2 次） |

### 12.3 因子交互分析

```
                     Trend Factor
                     ────────────
                         │
                         ▼
Donchian 突破 ──→ 入场信号 ──→ ATR 头寸规模 ──→ 开仓
     │                            │
     │                            ▼
     │                    动态止损 + 趋势因子调节
     │                            │
     │                     ┌──────┴──────┐
     │                     │             │
     │                     ▼             ▼
     │                价格有利       价格不利
     │                     │             │
     │                     ▼             ▼
     │              金字塔加仓       止损出场
     │                     │
     └─────────────────────┘
          (加仓放大收益)
```

**关键交互**：

1. **突破 + ATR 头寸**：突破信号决定方向，ATR 决定赌注大小——高波动（往往趋势末期）赌注小，低波动（往往趋势初期）赌注大
2. **止损 + 趋势因子**：止损宽度动态变化，防止强趋势中被震出
3. **加仓 + 止损**：加仓后止损距离不变（因为跟踪最高价），不会因为加仓而收紧止损
4. **Gap + 突破**：隔夜跳空可能在第二天开盘时让突破方向更极端，Gap 保护防止追高

### 12.4 因子贡献度估算

以下比例基于海龟系统 40 年历史数据和类似策略的学术研究估算：

| 因子 | Alpha 贡献 | 风险调整贡献 | 主要作用阶段 |
|------|-----------|-------------|-------------|
| Donchian 突破 | ~60% | ~40% | 趋势启动 |
| ATR 头寸规模 | ~15% | ~30% | 全程 |
| ATR 动态止损 | ~12% | ~20% | 持仓期 |
| 金字塔加仓 | ~8% | ~5% | 趋势确认期 |
| 趋势因子 | ~3% | ~3% | 持仓期 |
| Gap 保护 | ~2% | ~2% | 入场前 |

### 12.5 与经典因子的关系

| 经典因子 | V2 对应 | 差异 |
|---------|--------|------|
| 时间序列动量（TSMOM） | Donchian 突破 | 经典动量是 past N-day return，V2 用 20HL 通道突破而非累计收益 |
| 波动率管理（VolTargeting） | ATR 头寸规模 | 经典 VolTargeting 用已实现波动率调杠杆，V2 用 ATR 调每 Unit 手数 |
| 趋势强度（Trend Strength） | MA 趋势因子 | 经典方法用回归斜率或序列自相关，V2 用 MA 排列 + ATR 归一化 |

### 12.6 缺失的潜在因子

当前系统未使用但可能提升表现的因子：

| 因子            | 可能价值        | 实现复杂度           | 与现有因子相关性        |
| ------------- | ----------- | --------------- | --------------- |
| 成交量确认         | 突破时放量→信号更可靠 | 低（已有 volume 数据） | 低（Donchian 不看量） |
| 期限结构（升贴水）     | 远月升水→做空有利   | 高（需要多合约数据）      | 低               |
| 跨品种相关性        | 共振突破→信号更强   | 中               | 中               |
| 波动率趋势（ATR 斜率） | ATR 上升→趋势加速 | 低               | 高（与 ATR 本身相关）   |
