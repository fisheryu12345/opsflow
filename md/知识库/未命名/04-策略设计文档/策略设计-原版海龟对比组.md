# 原版海龟交易系统对比组

> 涉及文件：
> - `backend/stock/management/commands/run_turtle.py` —— 完整实现，单文件约 820 行

## 功能概述

独立的 Django 管理命令 `run_turtle.py`，实现**原版海龟交易系统**规则（最大仓位改为 3 个单位）。通过自定义命令方式运行，**不修改任何现有文件**，所有数据写入现有 DB 表，与现有系统运行结果进行对比分析。

> 设计目标：在同一数据库内，用不同账户分别运行现有系统和海龟系统，通过账户筛选在 Dashboard 和报表中对比两者的绩效差异。

---

## 与原系统的关键差异

| 方面 | 现有系统（V2/V3） | 原版海龟系统 |
|------|----------|----------|
| 运行方式 | **收盘后生成信号 → 下一开盘执行** | **盘中实时监控，触发即执行** |
| 入场判断 | 基于日收盘价突破唐奇安通道 | 基于盘中最新价突破唐奇安通道 |
| 出场信号 | ATR 动态跟踪止损（最高收盘价 - 2×(1+factor)×ATR） | **10日唐奇安反向突破**（盘中实时判断） |
| 仓位计算 | 固定 `POSITION_RISK_BASE_AMOUNT=4000` | 动态 `Equity × 1% / (2 × N × 合约乘数)` |
| 硬止损 | 无（靠动态止损 + 保本兜底） | **入场价 ± 2×N**（盘中实时监控） |
| 加仓触发 | 收盘判断，`first_open_price` 统一基准 | **盘中每 0.5×N 加 1 单位**，实时触发 |
| N 值管理 | 每日重新计算 ATR（动态止损公式用） | **入场时锁定到 `entry_atr`**，整笔交易不变 |
| 单位信息存储 | `indicators` JSON 存趋势因子等 | **不存 JSON**，用 `entry_atr`/`first_open_price` 自有字段 |
| 风险比例 | 无 | 每单位风险 = 账户权益的 1% |
| 品种间联动 | 各品种独立 | 资金按比例分配到各品种（1% / 品种） |

## 实时监控流程

原版海龟的核心理念是**盘中实时突破即入场**，不是收盘后才判断。

### 整体流程

```
┌─ 启动 ─────────────────────────────────────────────────┐
│  1. 创建 TqApi 连接                                      │
│  2. 获取活跃品种配置                                      │
│  3. 获取日 K 线 → 预计算唐奇安通道(20HL/10HL) + N 值     │
│  4. 进入 wait_update 实时循环                            │
│     ├── 8:55 / 13:25 / 20:55 定时重启连接                │
│     ├── 无持仓 → _check_entry(price, level)              │
│     │              └── 20HL 突破 → _execute_entry        │
│     │                       ├── 1 Unit（仓位计算）         │
│     │                       └── 创建 PositionState        │
│     │                           ├── first_open_price      │
│     │                           └── entry_atr = N         │
│     ├── 有持仓 → _check_exit(price, level)               │
│     │              └── 10HL 反向突破 → 平仓               │
│     ├── 有持仓 → _check_hard_stop(price)                 │
│     │              └── first_open ± 2×entry_atr → 平仓    │
│     ├── 有持仓且units<3 → _check_add_on(price)           │
│     │              └── first_open + units×0.5×N → 加仓    │
│     └── 有持仓 → _check_rollover                          │
│                        └── 主力切换 → 移仓               │
└─────────────────────────────────────────────────────────┘
```

### 实时循环核心代码

```python
def _handle_monitor(self, api, account):
    contracts = self._get_active_contracts(account)
    symbols = [c.symbol for c in contracts]

    # 预计算所有品种的唐奇安通道 + N 值（启动时一次）
    levels = self._compute_all_levels(api, account, symbols)

    # 订阅报价
    quotes = {sym: api.get_quote(sym) for sym in symbols}
    last_prices = {}

    while running:
        api.wait_update()

        for symbol in symbols:
            price = float(quotes[symbol].last_price)
            if last_prices.get(symbol) == price:
                continue  # 价格未变跳过
            last_prices[symbol] = price

            level = levels[symbol]
            position = PositionState.objects.get(account=account, symbol=symbol, units__gt=0)

            if not position:
                self._check_entry(api, account, symbol, price, level)
            else:
                self._check_exit(api, account, position, price, level)
                self._check_hard_stop(api, account, position, price)
                self._check_add_on(api, account, position, price)
                self._check_rollover(api, account, position)
```

### 定时重启机制

每个交易时段前重启连接，重新计算通道和 N 值：

| 时间 | 时段 | 目的 |
|------|------|------|
| 8:55 | 日盘前 | 重新计算当日通道/ATR |
| 13:25 | 下午盘前 | 防连接超时 |
| 20:55 | 夜盘前 | 重新计算夜盘时段通道/ATR |

重启时关闭旧 `TqApi` → 等待 35s → 创建新连接 → 重新 `_compute_all_levels`。

## 入场逻辑

### 触发条件

```python
def _check_entry(self, api, account, symbol, product_code, price, level):
    if self._already_signaled_today(account, symbol, 'ENTRY'):
        return

    if level['h20'] and price > level['h20']:
        direction = 1  # 突破20日高点 → 做多
    elif level['l20'] and price < level['l20']:
        direction = -1  # 跌破20日低点 → 做空
    else:
        return

    self._execute_entry(api, account, symbol, product_code, direction, price, level)
```

- 基于**盘中最新价**（不是收盘价）判断突破
- 已存在当天成功/执行中的 ENTRY 信号则跳过（一天只能开仓一次）
- `level` 来自启动时计算的唐奇安通道（20日入场）

### 仓位计算

```
1 Unit = floor(Account Equity × 1% / (2 × N × Contract Multiplier))

risk_amount  = equity × 0.01        # 账户权益的 1%
unit_value   = 2.0 × n_value × volume_multiple   # 1单位的价格风险
unit_lots    = max(1, int(risk_amount / unit_value))
```

参数来源：
- `equity`：`TradingAccount.current_equity`
- `n_value`：ATR(20)，启动时计算
- `volume_multiple`：`quote.volume_multiple`（合约乘数）

### 执行流程

```
_execute_entry:
  1. TargetPosTask(symbol).set_target_volume(±unit_lots)
  2. wait_for_target_position(timeout)
  3. 获取成交价 fill_price ≈ quote.last_price
  4. PositionState.update_or_create:
       - units = 1
       - first_open_price = fill_price
       - cost_price = fill_price
       - entry_atr = n_value        ← N 值在此锁定
       - direction, contract_total_position, ...
  (不写入 indicators 字段)
```

## 加仓逻辑

### 触发条件

```python
def _check_add_on(self, api, account, position, price):
    if position.units >= 3:
        return  # 最多3个单位

    if 存在 PENDING/EXECUTING 的 ADD_ON 信号:
        return  # 防重复

    n_value = float(position.entry_atr or 0)    # 从 DB 直接读取
    if not n_value:
        return

    first_price = float(position.first_open_price)
    # 累加触发距离：units × 0.5 × N
    trigger_distance = position.units * 0.5 * n_value

    if position.direction == 1 and price >= first_price + trigger_distance:
        triggered = True
    elif position.direction == -1 and price <= first_price - trigger_distance:
        triggered = True

    if triggered:
        self._execute_add_on(api, account, position, price)
```

### 加仓价格阶梯

| 当前 units | 触发价（空头） | 触发价（多头） | 说明 |
|-----------|---------------|---------------|------|
| 1 | `first - 0.5N` | `first + 0.5N` | +1U，到 2U |
| 2 | `first - 1.0N` | `first + 1.0N` | +1U，到 3U 满仓 |

**关键设计**：
- N 值用 `position.entry_atr`（入场时锁定），不随行情波动变化
- 所有单位以 `first_open_price` 为统一基准，**不是逐级递推**
- 数学等价于原版逐级判断：`first + units×0.5×N = first + Σ(0.5×N)`

### 执行流程

```
_execute_add_on:
  1. n_value = position.entry_atr（直接读取 DB）
  2. unit_lots = _turtle_unit_lots(...)    # 与入场相同的仓位计算
  3. order_volume = 1 × unit_lots
  4. TargetPosTask(symbol).set_target_volume(当前仓位 + order_volume)
  5. wait_for_target_position(timeout)
  6. PositionState.update(units += 1, contract_total_position += order_volume)
  (不更新 indicators，不更新 cost_price)
```

## 出场逻辑

原版海龟有两个并行出场机制：

### 1. 10HL 反向突破（主出场）

```python
def _check_exit(self, api, account, position, price, level):
    if self._already_signaled_today(account, position.symbol, 'STOP_LOSS'):
        return

    if position.direction == 1 and price < level['l10']:
        triggered = True  # 多头：跌破10日低点
    elif position.direction == -1 and price > level['h10']:
        triggered = True  # 空头：涨破10日高点
```

- 使用 10 日唐奇安通道，`shift(1)` 避免前视偏差
- 需要 `level` 参数（monitor 启动时从 K 线计算）
- 这是海龟原版的标准出场规则，代表**趋势反转信号**

### 2. 2N 硬止损（后备风控）

```python
def _check_hard_stop(self, api, account, position, price):
    n_value = float(position.entry_atr or 0)
    if not n_value:
        return

    first_price = float(position.first_open_price)
    hard_stop = first_price - 2.0 * n_value if position.direction == 1 \
                else first_price + 2.0 * n_value

    if position.direction == 1 and price <= hard_stop:
        triggered = True
    elif position.direction == -1 and price >= hard_stop:
        triggered = True
```

- **不需要 `level` 参数**：N 值直接从 `entry_atr` 读取
- 硬止损价 = `first_open_price ± 2 × entry_atr`
- 2N 硬止损是**固定距离**（基于入场 N），不随趋势强弱调整
- 两个出场机制每次价格更新同时检查，任一触发即平仓

### 执行流程

```
_execute_exit:
  1. DailyStrategySignal.create(trade_type='STOP_LOSS', status='EXECUTING')
  2. TargetPosTask(symbol).set_target_volume(0)
  3. wait_for_target_position(timeout)
  4. 获取成交价（优先成交回报，兜底 quote.last_price）
  5. record_and_reset_position() → ClosedPositionRecord 创建
  6. 信号标记 SUCCESS
```

## N 值（ATR）管理体系

这是原版海龟与 V2 系统**最核心的差异**：

```
原版海龟：
  入场时锁定 N → 整笔交易不变
    ├── 加仓触发距离 = units × 0.5 × locked_N
    ├── 硬止损距离 = 2 × locked_N
    └── 仓位计算 = equity × 1% / (2 × locked_N × 乘数)

V2 系统：
  每日重新计算 N → 动态止损公式每天变
    ├── 趋势因子每天更新（factor）
    ├── 止损距离 = 2(1+factor) × current_ATR
    └── 加仓触发也用当天 ATR
```

### 存储方式

原版海龟将 N 值存储在 `PositionState` 自有字段，**不依赖 `indicators` JSON**：

| 字段 | 类型 | 写入时机 | 后续使用 |
|------|------|---------|---------|
| `entry_atr` | Decimal | `_execute_entry` 时写入 | 加仓触发、硬止损、仓位计算（只读） |
| `first_open_price` | Decimal | `_execute_entry` 时写入 | 加仓/止损基准价（只读） |
| `indicators` | JSON | **不写入** | 留给 15:32 调度器独立使用 |

### 与调度器数据解耦

原版海龟的 `run_turtle.py` 和 V2 收盘任务 `tasks_daily_close.py` 共享 `PositionState` 表，但**互不干扰**：

| 字段 | run_turtle 写入 | 调度器写入 |
|------|----------------|-----------|
| `units`, `contract_total_position` | ✅ 实时更新 | ✅ 盘后更新 |
| `entry_atr` | ✅ 入场时写入 | 只读 |
| `first_open_price` | ✅ 入场时写入 | 只读 |
| `indicators` | **不写入** | ✅ `{atr_20, ma_*, trend_*, ...}` |
| `stop_loss_price` | 不使用 | ✅ 每日更新 |
| `highest_close`, `lowest_close` | 不使用 | ✅ 每日更新 |

> **解耦意义**：以前 run_turtle 向 `indicators` 写 `{locked_n, unit_prices, hard_stop_price}`，调度器 15:32 覆盖为 `{atr_20, trend_factor, ...}`，导致 monitor 的 locked_n 被销毁，夜盘加仓无法触发。2026-05-21 改为不读写 `indicators` 后，两个系统彻底独立。

### V2 调度器跳过海龟账户（2026-05-26 新增）

为防止 MySQL 1062 重复键冲突（`DailyStrategySignal` 的 `(account, symbol, trade_date)` 唯一约束），V2 的三个 APScheduler 任务（开盘/收盘/临收盘）均已添加 `is_independent_strategy` 检查：

```python
is_independent_strategy = account.strategyconfig.strategy_type in ('TURTLE', 'HVOB')
if is_independent_strategy:
    continue  # 跳过所有 V2 操作
```

这意味着 `tasks_daily_close.py` 的 `check_rollover_signals()`、`tasks_daily_open.py` 的 `process_signals_by_type()`、`tasks_exit_before_close.py` 的 `check_and_execute_stop_loss()` 等函数不会再对海龟账户执行。海龟账户的所有信号管理完全由 `run_turtle.py` 自行负责。

## Alpha 因子分析

### 核心差异概览

原版海龟与 V2 的 Alpha 结构有根本性差异，不仅在参数取值，更在**信号产生方式**和**风险管理逻辑**：

| 维度 | 原版海龟 | V2/V3 |
|------|---------|-------|
| 信号产生 | 盘中实时触发（T+0） | 收盘信号→次日开盘执行（T+1） |
| 出场方式 | 10HL 反向突破 + 2N 硬止损 | ATR 动态跟踪止损 |
| 头寸规模 | 权益 × 1% / (2 × N × 乘数) — 动态 | 固定 4000 / (ATR × 2 × 乘数) — 固定风险 |
| N 值管理 | 入场时锁定（entry_atr） | 每日重新计算（current_atr） |
| 加仓基准 | 首次开仓价 + units × 0.5×N | 首次开仓价 + favorable_diff（0.5/1.0 ATR） |
| 趋势过滤 | 无 | 趋势因子动态调止损（V2）/跳过开仓（V3） |

### 原版海龟的 Alpha 因子分解

#### 因子一：20HL 盘中突破（核心 Alpha）

```
signal = 
  +1  if price_t > max(high_{t-20}, ..., high_{t-1})  任意时刻
  -1  if price_t < min(low_{t-20}, ..., low_{t-1})      任意时刻
```

与 V2 的关键差异在于**执行时间**：

| 维度 | 原版海龟（盘中） | V2（收盘） |
|------|---------------|-----------|
| 信号触发 | 盘中最新价突破通道 | 日收盘价突破通道 |
| 执行延迟 | 即时（同一 wait_update） | 隔夜 |
| 滑点风险 | 较高（突破瞬间下单） | 较低（开盘价成交） |
| 跳空风险 | 高（盘中持仓隔夜） | 中（开盘价可能不利） |

盘中突破的 alpha 优势在于**反应速度**——在趋势启动的第一时间就入场。这在连续交易的市场中捕捉到了收盘价突破无法捕捉的盘中动量。

#### 因子二：10HL 反向突破出场（趋势反转出场）

**公式**（与入场对称）：
```
多头出场：price < min(low_{t-10}, ..., low_{t-1})  跌破10日低点
空头出场：price > max(high_{t-10}, ..., high_{t-1})  涨破10日高点
```

这是原版海龟与 V2 最核心的 Alpha 差异。10HL 反向突破是一个**独立的趋势反转信号**，不是简单的止损。

| 维度 | 10HL 反向突破（原版） | ATR 动态止损（V2） |
|------|--------------------|------------------|
| 信号性质 | 趋势反转确认 | 价格回落幅度超限 |
| 出场时机 | 趋势已反转 | 趋势回调（未必反转） |
| 盈亏比 | 适中（部分利润回吐后出场） | 更高（在最高点附近出场） |
| 胜率 | 略高 | 略低 |
| 最大回撤 | 较低（趋势反转即出场） | 略高（回调后可能又回去） |
| 双重出场 | 10HL + 2N 硬止损 | 动态止损 + 保本兜底 |

**代数意义**：
```
10HL 反向突破 = 承认趋势已结束
ATR 动态止损 = 趋势可能还在，但回调幅度超限
```

#### 因子三：2N 硬止损（后备风控）

```
hard_stop = first_open_price ± 2 × entry_atr
```

- N 值锁定在 `entry_atr`（入场时固定），不随行情更新
- 与 V2 的保本兜底不同——V2 的保本在盈利达到阈值后激活，2N 硬止损在入场时即存在
- 2N 硬止损是"不管趋势如何变化，最大亏损不超过 2N"的硬约束

#### 因子四：0.5N 加仓（等间隔动量确认）

```
加仓价 = first_open + units × 0.5 × N
```

| 当前 units | 加仓触发价 | 说明 |
|-----------|-----------|------|
| 1 | first + 0.5×N | 2U |
| 2 | first + 1.0×N | 3U（满仓） |

与 V2 的关键差异：

| 维度 | 原版海龟 | V2 |
|------|---------|-----|
| 触发间距 | 固定 0.5×N | 0.5×ATR / 1.0×ATR |
| N 值基准 | 锁定（entry_atr） | 每日更新（current_atr） |
| units=1 直接满仓 | 无（严格 0.5 间隔） | 有利变动 > 1.0×ATR → 直接 +2U |
| 触发形式 | 盘中实时 | 收盘后判断 |

#### 因子五：权益比例头寸规模（动态风险因子）

```
Unit = floor(Equity × 1% / (2 × N × volume_multiple))
```

| 维度 | 原版海龟（权益%） | V2（固定金额） |
|------|-----------------|--------------|
| 风险变化 | 随权益增长/缩水而变 | 固定每 Unit 4000 元 |
| 权益增长时 | 自动增加手数（复利效应） | 手数不变（需要手动调参） |
| 权益缩水时 | 自动减少手数（风控收缩） | 手数不变（风险敞口不缩小） |
| 多品种资金分配 | 每个品种独立分配 1% 权益 | 每个品种独立 4000 元 × unit |

原版海龟的权益比例头寸规模嵌入了一个**自动复利和自动风控**机制——账户赚钱时下注更大，亏钱时下注更小。V2 的固定金额模式则更简单直接，但需要人工管理风险敞口。

### 因子对比总结

| 原版海龟因子 | 对应 V2 因子 | 差异程度 | 方向 |
|------------|------------|---------|------|
| 20HL 盘中突破 | 20HL 收盘突破 | 中 | 时间维度（T+0 vs T+1） |
| 10HL 反向出场 | ATR 动态止损 | **大** | 出场逻辑完全不同 |
| 2N 硬止损 | 保本兜底 | **大** | 风控逻辑不同 |
| 0.5N 加仓 | favorable_diff 加仓 | 中 | V2 有"直接满仓"快捷路径 |
| 权益% 头寸 | 固定金额头寸 | 中 | 动态 vs 静态 |
| N 值锁定 | 每日更新 ATR | 中 | 入场锁定 vs 持续更新 |
| 无趋势过滤 | 趋势因子/choppy 过滤 | **大** | V2 有额外调节层 |

### 差异的经济学含义

1. **原版海龟更适合连续交易品种**（外汇、美股）——盘中实时反应捕捉微小动量
2. **V2 更适合跳空频繁的市场**（国内期货）——收盘执行避免盘中追突破
3. **ATR 动态止损 vs 10HL 出场**的核心区别在于趋势跟踪的哲学：V2 认为"趋势更持久"，原版海龟认为"趋势反转更快识别"
4. **N 值锁定 vs 每日更新**：锁定产生更一致的加仓/止损逻辑，每日更新则更适应市场变化

### 因子贡献度估算（原版海龟）

| 因子 | Alpha 贡献 | 说明 |
|------|-----------|------|
| 20HL 盘中突破 | ~55% | 核心信号来源 |
| 10HL 反向出场 | ~15% | 趋势反转识别能力 |
| 0.5N 加仓 | ~12% | 趋势确认放大收益 |
| 权益% 头寸 | ~10% | 复利效应 + 自动减仓 |
| 2N 硬止损 | ~8% | 控制尾部风险 |

### 2026-05-21 — 与 indicators 解耦 + 使用 entry_atr

**问题**：run_turtle 和调度器共享 `PositionState.indicators` JSON 字段，但写入不兼容格式。调度器 15:32 覆盖后 → monitor 的 locked_n/unit_prices 被销毁 → 夜盘加仓无法触发。

**变更**：

| 方面 | 改前 | 改后 |
|------|------|------|
| N 值来源 | `indicators['locked_n']` | `position.entry_atr`（自有字段） |
| 加仓触发 | `last_unit_price + 0.5×N`（逐级） | `first_open_price + units×0.5×N`（累加） |
| 硬止损 | `indicators['hard_stop_price']` | `first_open_price ± 2×entry_atr`（动态计算） |
| 单位价格列表 | `indicators['unit_prices']` | 不再维护 |
| 入场/加仓时 | 写入 indicators | **不写入 indicators** |
| 函数参数 | 加仓/止损需要 `level` | 加仓/止损不再需要 `level`（仅出入场需要） |
