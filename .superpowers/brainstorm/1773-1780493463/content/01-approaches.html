前端网关条件编辑 UI — 三种方案对比

当前问题：条件表达式只能手写原始 ${node_1.cpu} > 80 文本，无提示、无校验、无结构

方案 A：极简增强版

保留现有 PropertyPanel 文本区域 + 增强辅助功能

label
expr
{x}
${node_1.cpu} ${node_1.mem} ${global.env} + 点击插入变量
✓ 校验通过：node_1.cpu 存在，类型 number

优势

  • 改动最小，2-3 天
  • 不改变已有交互流程
  • 变量按钮弹出选择器快速插入

局限

  • 仍然是自由文本输入
  • 复杂条件容易写错语法
  • 无运算符辅助选择

方案 B：条件构建器（推荐）

结构化的行式条件编辑器，变量 + 运算符 + 值三段式

1
2
> < == != contains regex
3 (numeric)
生成：${node_1.cpu} > 80

优势

  • 零学习成本，点选即可
  • 类型感知（数值显示数字键盘提示）
  • 支持 AND/OR 多条件组合
  • 实时校验 + 表达式预览

局限

  • 实现工作量大（5-7 天）
  • 非常复杂的表达式可能不如文本灵活

方案 C：混合模式

简单条件用快捷按钮，复杂条件回退到文本编辑器

${_result} == True ${_result} == False 自定义...
条件表达式：
⚠ ${node_2.mem} 不存在

优势

  • 常用条件一键插入（成功/失败/结果判等）
  • 复杂表达式保留灵活性
  • 工作量适中（3-4 天）

局限

  • 体验不够统一
  • 多条件组合仍然靠手写
