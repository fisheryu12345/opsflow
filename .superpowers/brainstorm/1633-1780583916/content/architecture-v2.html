OpsFlow 架构决策 · 基于你的选择

CMDB → Neo4j | 单体优先 | 阶段化演进

✅ 已确认：CMDB 采用 Neo4j

CMDB 天然适合图数据库：

  • 拓扑建模 — 业务 → 集群 → 模块 → 主机 → 进程，天然是图结构
  • 影响分析 — "这台主机挂了会影响哪些服务？"在 Neo4j 里是 O(1) 级别的遍历
  • 路径发现 — 调用链追踪、网络依赖分析，图数据库比 MySQL JOIN 快 10-100×
  • 自定义模型 — Neo4j 的 schema-free 特性天然适配 CMDB 模型动态扩展的需求

待确认：三大架构决策

决策 1：CMDB 的技术架构怎么搭？

A

neomodel (Python OGM)

类 Django ORM 的图对象映射库，用 Python 定义节点/关系，与 Django 集成最好

pip install neomodel · 支持 Django Signal · 熟悉度高
B

py2neo (底层驱动)

更灵活的 Cypher 查询构建，适合复杂图查询，但需要手写 Cypher

更灵活 · 学习曲线高 · 完全控制 SQL/Cypher
C

Django + Neo4j 双数据源

CMDB 业务数据在 Neo4j，关联的 RBAC/审批/审计在 MySQL，Django 配置双数据库路由

最佳实践 · 各取所长 · 需要 router 配置

决策 2：ITSM 工单是否也采用图数据存储？

A

ITSM → MySQL 纯关系模型

工单/流程/SLA 是事务性数据，MySQL 更合适。状态机引擎 + 流程模板

ACID 事务保障 · Django ORM 直接使用 · 成熟度高
B

工单主体 MySQL + CMDB 关联走 Neo4j

工单表中的资产/配置项字段存储 Neo4j 节点 ID，查询时跨数据源联合

灵活关联 · 复杂度中等

决策 3：第一阶段(Phase 1)的边界是什么？

A

最小可用 CMDB

只做 CMDB 核心模型（业务/集群/模块/主机/进程）、拓扑视图、基础 CRUD API。先让 CMDB 跑起来，其他模块后续接入

~2周 · 最聚焦
B

CMDB + ITSM 工单绑定

CMDB 上线同时搭建工单系统，工单能关联 CMDB 配置项，事件管理走通"发现→提单→处理→关闭"闭环

~4周 · 闭环价值
C

CMDB + ITSM + 监控告警接入

三条线一起：CMDB 定义资产、工单管流程、监控发现即告警，告警自动创建事件工单

~8周 · 高风险高回报
