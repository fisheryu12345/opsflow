两条采集路径的职责划分

🔵 Prometheus + Exporter

拉模式 (Pull)
• 基础设施指标 (CPU/内存/磁盘/网络)
• 中间件指标 (MySQL/Redis/Nginx)
• K8s 集群指标 (kube-state-metrics)
• 网络探测 (blackbox-exporter)
• 自定义 exporter

存储：Prometheus TSDB (本地/远端)
保留策略：短期高精度 (15d)

🟢 Telegraf + InfluxDB

推模式 (Push)
• 业务自定义指标 (API 埋点)
• 网络设备指标 (SNMP/Syslog)
• 应用性能指标 (JMX/Kafka)
• 云厂商指标 (AWS/Azure/阿里云)
• 自定义脚本采集

存储：InfluxDB
保留策略：长期抽样 (90d+)

OpsFlow 平台的监控集成点

① 告警消费
OpsFlow 通过 Grafana Alerting API 读取告警事件 → 创建 ITSM 事件工单 → 通知负责人
② 数据源管理
OpsFlow 管理监控目标(Target)注册，自动同步 CMDB 主机到 Prometheus SD 或 Telegraf 配置
③ 仪表盘嵌入
OpsFlow 页面通过 iframe/Grafana API 嵌入 CMDB 资产的实时监控面板

告警→工单联动流程

Grafana Alert 触发
    ↓
Grafana Webhook / Alertmanager Webhook
    ↓
OpsFlow webhook receiver (复用 ops_webhook_config)
    ↓
告警收敛引擎: 去重 + 聚合 + 抑制 (Redis 缓存)
    ↓
关联 CMDB: 根据标签/hostname/IP 找到受影响资产
    ↓
确定影响范围 & 影响业务
    ↓
┌─── 自动模式 ─────────────────────────┐
│  创建事件工单 (Incident)              │
│  SLA 开始计时                        │
│  通知负责人 (WebSocket/企微/邮件)    │
└──────────────────────────────────────┘

⚡ 确认：目前的监控方案

  • 采集：Prometheus + Exporter (基础设施) + Telegraf + InfluxDB (业务/网络设备)
  • 可视化 + 告警：Grafana (双数据源: Prometheus + InfluxDB)
  • OpsFlow 集成：通过 Grafana Alerting API / Webhook 消费告警 → 联动 ITSM 工单
  • 告警收敛、通知路由、资产关联由 OpsFlow 平台管理
