ITSM 分派优化 — 方案对比

基于现有体系（ServiceCategory + Role + DistributeType），三种实现路径

1

轻量增强 — 优先完善现有机制

改动最小，复用现有模型

  • 手动分派写入 TicketStatus.processors，真正生效
  • Designer 暴露 distribute_type + ORGANIZATION 类型
  • category → service_group 映射配置在 ServiceCategory 加字段
  • 升级用现有 SlaPolicy.escalate_minutes
  • 转派记录到 ticket.state_history + meta.transfer_log

优点

  • 1-2 天可完成
  • 无 migration
  • 兼容已有工单

缺点

  • 升级只有单级
  • 无排班/负载均衡
2

中等增强 — 新增 SkillGroup + OnDuty

推荐，覆盖完整场景

  • 新增 SkillGroup（技能组）+ OnDutySchedule（排班）模型
  • ServiceCategory 关联默认 SkillGroup
  • 新引擎: category → skill_group → on-duty member 自动分派
  • EscalationLevel 模型：L1→L2→L3 可配置阈值+通知
  • TransferLog 模型记录完整转派审计
  • 手动分派仍然是"覆盖"机制

优点

  • 完整 IT 运维场景覆盖
  • 可扩展排班/负载
  • 多级升级

缺点

  • 3-5 天开发
  • 需要 migration
  • 需要初始数据配置
3

全面体系 — +值班日历+负载均衡+自动路由引擎

大而全，适合中大型运维团队

  • 方案2 全部 + 值班日历(周/月视图)
  • 实时负载均衡（按待办数量/工单权重）
  • 自动路由引擎（RuleEngine: IF category=X AND priority=P1 THEN group=A）
  • SLA 联动升级（超时自动升级+通知）
  • Dashboard 展示团队负载/分派统计

优点

  • 最自动化
  • 适合大规模团队

缺点

  • 1-2 周开发
  • 配置复杂
  • 可能过度设计
